# Каноническая Архитектура Agentic UX Engine (Aura)

⚠️ **Важно:**
Ассистент Aura является событийной UX-функцией.
Он не является чат-ботом, не ведёт диалог и не хранит память.
Все решения принимаются MetaAgent на основе событий опыта.

---

## 1. Архитектурная Иерархия

Система построена на строгом разделении ответственности (Separation of Concerns):

1.  **Agents (System V2.0)** → *Генераторы Контента*. Определяют, **что** показать (товары, анализ, подсказки). Не имеют доступа к состоянию приложения.
2.  **MetaAgent (System V3.0)** → *Оркестратор Опыта*. Определяет, **когда** показать контент, основываясь на State Machine (FSM). Управляет переходами между состояниями (`IDLE` → `BROWSING` → `AR_ACTIVE`).
3.  **AssistantRenderer** → *Визуализатор*. Определяет, **как** отобразить контент, полученный от агентов, в зависимости от текущего режима.
4.  **UI Components** → *Представление*. «Глупые» компоненты (`ProductCarousel`, `ObjectCard`), рендерящие данные.

---

## 2. Визуализация и UX (Frontend)

### 2.1 AssistantRenderer & ChatBubble
*Ранее: ChatWidget*

`ChatBubble` — это **опциональный визуальный режим** ассистента, активный исключительно в состояниях:
*   `BROWSING` (Подбор)
*   `SELECTING` (Уточнение)
*   `POST_AR_REFLECTION` (Рефлексия после AR)

В режиме `AR_ACTIVE` (погружение) ассистент полностью скрыт, чтобы не нарушать иммерсивность (Quiet UX).

### 2.2 Skeletons (Ghost UI)
Визуализируют **процесс обработки запроса** агентами.
*   **Назначение:** Индикация работы системы без блокировки интерфейса.
*   **Типы:**
    *   `AnalysisSkeleton` — обработка визуального контекста (фото).
    *   `ProductCarouselSkeleton` — поиск кандидатов в каталоге.
*   **Принцип:** Не обещают конкретного результата (ответ может быть пустым), но поддерживают динамику опыта.

### 2.3 Visual Components (Generative UI)
Агенты декларируют компоненты, AssistantRenderer их инстанцирует:
*   **ProductCarousel:** Витрина объектов. Реализует паттерн **Click-to-Real**: выбор объекта генерирует событие `VIEW_OBJECT`, которое переводит MetaAgent в состояние просмотра.
*   **AnalysisResult:** Оверлей с результатами работы Vision Agent.
*   **ObjectCard:** Единица контента с `aiReasoning` (обоснованием выбора).

### 2.4 Action Chips (MetaEvents)
*Ранее: Suggestion Chips*

Это не текстовые подсказки для чата, а декларативные **триггеры событий**.
*   **Механика:** Нажатие на чип генерирует структурированное событие (MetaEvent), которое обрабатывается MetaAgent.
*   **Пример:**
    *   UI: Кнопка [Темнее]
    *   Event: `{ type: "REFINE_STYLE", payload: { modifier: "darker" } }`

## 3. Реестр Агентов (Backend Deep Dive)

Агенты (Server Actions) работают в режиме "Request-Response" без сохранения истории.

### 3.1 SupervisorAgent (Маршрутизатор)
*   **Input:** Входящее событие (текст, фото, MetaEvent).
*   **Logic:** Классифицирует интент и маршрутизирует запрос.
*   **Output:** `AgentResponse` (Текст + Компонент + Chips).
*   **Особенность:** Не ведет диалог. Передает управление специализированным агентам.

### 3.2 CatalogAgent (Куратор)
*   **Input:** Явно сформированный `Context Object` (приходит от Supervisor).
    *   Пример контекста: `{ style: "japandi", excludedIds: [...] }`.
*   **Logic:**
    1.  **Search:** Запрос к БД по критериям контекста.
    2.  **Curation:** Выбор лучших визуальных совпадений.
*   **Refinement:** "Уточнение" происходит не через чтение истории чата, а через модификацию входящего Context Object супервизором (напр. добавление ID в `excludedIds`).
*   **Output:** Список объектов (`products`).

### 3.3 VisionAgent (Инспиратор)
*   **Input:** Изображение (Base64).
*   **Output:** Дизайнерский контекст (Стиль, Настроение, Палитра, Координаты).
*   **Role:** Создает "атмосферную" базу для работы CatalogAgent. Не является инструментом замера помещения.

### 3.4 GuardianAgent (Цензор)
*   **Role:** Обеспечивает Brand Safety на выходе.
*   **Logic:** Статический анализ и санитайзинг контента (удаление цен, скидок, спама) перед отправкой на клиент. Гарантирует "Quiet UX".

---

## 4. Поток Данных (Canonical Data Flow)

1.  **User Event:** Пользователь совершает действие (загрузка фото / выбор чипа / ввод).
2.  **MetaAgent:** Обрабатывает событие, обновляет State (`BROWSING` и т.д.).
3.  **Server Action:** Вызывается `SupervisorAgent` с текущими данными.
4.  **Agent Processing:** Агенты (Vision/Catalog) обрабатывают запрос изолированно.
5.  **Rendering:** `AssistantRenderer` получает ответ и обновляет UI (показывает Bubble, Carousel или Chips).
6.  **UX Loop:** Действие пользователя обновляет состояние опыта, замыкая круг.

---

## 5. Статус Канонизации
*   **Agents V2.0:** Интегрированы как чистые функции (Stateless).
*   **MetaAgent V3.0:** Является единственным владельцем состояния (State Owner).
*   **ChatBubble:** Реализован как режим отображения, а не центральный хаб.
*   **History:** Удалена. Система реагирует только на «Здесь и Сейчас».
